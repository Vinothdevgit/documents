— The Flow
01
Server Startup
Once at launch
Windsurf launches │ ▼ Spawns child process $ node my-server.js │ ▼ StdioServerTransport created stdin/stdout pipes connected │ ▼ Handshake Windsurf ──► "initialize" Server ──► "here are my capabilities"
What happens
  • Windsurf spawns your server as a child process
  • stdio pipes are established (stdin + stdout)
  • initialize handshake confirms protocol version
  • tools/list discovery — Windsurf learns your tools
  • Server goes IDLE, waiting on stdin
02
You Type a Prompt
User input
You type: "What is the weather in Singapore?" │ ▼ Windsurf sends your text to the LLM
03
LLM Decides to Use a Tool
LLM reasoning
LLM receives: - message: "What is the weather in Singapore?" - tool list: [get-weather, ...] │ ▼ LLM THINKS: "I have a tool called get-weather. It needs { city: string }. I should call it → city = Singapore" │ ▼ LLM responds with tool call (NOT plain text): { "tool": "get-weather", "arguments": { "city": "Singapore" } }
Key insight

The LLM doesn't answer yet. It recognises a tool should be used and emits a tool call instead of a human response. Windsurf intercepts this and routes it to your MCP server.

04
Windsurf → MCP Server via stdio
stdio + JSON-RPC
Windsurf writes to stdin: { "jsonrpc": "2.0", "method": "tools/call", "params": { "name": "get-weather", "arguments": { "city": "Singapore" } }, "id": 5 } │ travels through stdin PIPE ▼ StdioServerTransport reads it ▼ transport.onmessage fires ◄── the wire! ▼ McpServer.handleMessage() receives it
stdio in action

This is where the stdin pipe does its job. Windsurf writes a newline-delimited JSON-RPC message. The transport reads it, fires the wired handler, and your MCP server takes over.

05
Zod Validates the Input
Type safety
McpServer got: { city: "Singapore" } │ ▼ Zod schema runs: z.object({ city: z.string() }) .parse({ city: "Singapore" }) │ ├── ✅ Valid → proceeds to handler └── ❌ Invalid → error returned (API never called)
Why Zod matters

Zod acts as the bouncer. It validates arguments before any code runs. It also auto-generates the JSON Schema that Windsurf uses during tool discovery — so the LLM knows exactly what shape to pass.

06
Tool Handler Calls Backend API
Your code runs
Zod passed ✅ │ ▼ server.tool("get-weather", ..., async ({ city }) => { // YOUR CODE — calls the real backend const response = await fetch( `https://api.weather.com/v1?city=${city}` ) const data = await response.json() return { content: [{ type: "text", text: `Weather in ${city}: ${data.temp}°C` }] } }) │ ▼ fetch() ──► Backend API responds
This is your code

Everything inside the handler is written by you. You can call any REST API, query a database, read files — anything Node.js can do. The result is passed back up through the chain.

07
Response Travels Back to You
Full circle
Backend API returns data │ ▼ Handler returns result to McpServer │ ▼ McpServer wraps as JSON-RPC response writes to stdout │ travels through stdout PIPE ▼ Windsurf reads from stdout sends tool result back to LLM │ ▼ LLM generates human-friendly answer │ ▼ You see the response in Windsurf ✅
Full circle

The raw API data flows back through stdout → Windsurf → LLM. The LLM then composes a natural language answer using the tool result. You only ever see the final, human-readable reply.
